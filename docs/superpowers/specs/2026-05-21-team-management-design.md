# 团队管理功能完善设计文档

**日期:** 2026-05-21
**模块:** 团队邀请管理 + 团队管理
**目标:** 完善团队邀请机制（邀请码）和团队管理功能（角色变更、成员移除、团队设置）

## 概述

当前系统已有基础团队功能（数据库表、基础 API、团队列表/详情页面），但缺少完整的邀请流程和团队管理能力。本设计分两个阶段实现：

**阶段一：邀请码机制**
- 可刷新邀请码 + 直接加入模式
- 团队详情页邀请码展示
- 独立"加入团队"页面

**阶段二：团队管理功能**
- 角色变更（Owner/Admin 可操作）
- 移除成员（Owner/Admin 可操作）
- 团队设置（重命名、删除）

## 阶段一：邀请码机制

### 1. 数据库变更

#### teams 表新增字段

```sql
ALTER TABLE teams ADD COLUMN invite_code TEXT UNIQUE;
ALTER TABLE teams ADD COLUMN invite_code_updated_at TIMESTAMP WITH TIME ZONE;
```

#### 删除旧邀请表

```sql
DROP TABLE team_invites;
```

#### 数据迁移

为现有团队生成初始邀请码：

```sql
UPDATE teams
SET invite_code = UPPER(SUBSTRING(MD5(RANDOM()::TEXT), 1, 8)),
    invite_code_updated_at = NOW()
WHERE invite_code IS NULL;
```

**邀请码生成规则：**
- 8位大写字母 + 数字组合
- 使用 `UPPER(SUBSTRING(MD5(RANDOM()::TEXT), 1, 8))`
- PostgreSQL UNIQUE 约束保证唯一性（绝对不会重复）
- 避免混淆字符（不使用 0/O, 1/I/L）

### 2. API 设计

#### a) 刷新邀请码

```
POST /api/teams/{teamId}/invite-code
权限: owner/admin
请求体: 无
返回: { inviteCode: string, updatedAt: string }

流程:
1. 验证用户权限（必须是 owner 或 admin）
2. 生成新的随机邀请码
3. 更新 teams 表的 invite_code 和 invite_code_updated_at
4. 返回新邀请码
```

#### b) 验证邀请码

```
GET /api/teams/invite-code?code={inviteCode}
权限: 登录用户
返回: { teamId: string, teamName: string } 或错误信息

流程:
1. 验证邀请码是否存在对应团队
2. 返回团队基本信息（预览）
3. 不执行加入操作，仅验证有效性
```

#### c) 加入团队

```
POST /api/teams/join
权限: 登录用户
请求体: { inviteCode: string }
返回: { teamId: string, teamName: string, role: 'member' }

流程:
1. 验证邀请码是否存在对应团队
2. 检查用户是否已是该团队成员
3. 添加用户为 member 角色
4. 返回团队信息
```

## 阶段二：团队管理功能

### 1. API 设计

#### a) 角色变更

```
PUT /api/teams/{teamId}/members/{userId}/role
权限: owner/admin（但不能修改 owner 的角色）
请求体: { role: 'owner' | 'admin' | 'member' }
返回: { member: TeamMember }

流程:
1. 验证操作者权限（必须是 owner 或 admin）
2. 验证目标成员是否存在
3. 如果操作者是 admin，禁止修改 owner 的角色
4. 更新 team_members.role
5. 返回更新后的成员信息
```

**权限规则:**
- Owner 可以修改任何人角色（包括将成员提升为 Owner）
- Admin 可以修改 Admin 和 Member 角色，但不能修改 Owner
- Member 无权修改任何角色

---

#### b) 移除成员

```
DELETE /api/teams/{teamId}/members/{userId}
权限: owner/admin（但不能移除 owner/admin）
返回: { success: true }

流程:
1. 验证操作者权限
2. 验证目标成员角色（不能移除 owner 或 admin）
3. 删除 team_members 记录
4. 返回成功状态
```

**权限规则:**
- Owner 可以移除 Admin 和 Member
- Admin 可以移除 Member，但不能移除 Owner 或 Admin
- Member 无权移除任何人
- Owner 不能移除自己（需要先转让 Owner 角色）

---

#### c) 重命名团队

```
PUT /api/teams/{teamId}
权限: owner/admin
请求体: { name: string }
返回: { team: Team }

（此接口已存在，确认权限控制正确）
```

---

#### d) 删除团队

```
DELETE /api/teams/{teamId}
权限: 仅 owner
返回: { success: true }

（此接口已存在，确认权限控制正确）
流程:
1. 验证用户是 team owner
2. 删除 teams 记录（CASCADE 自动删除 team_members + team_prompts）
3. 返回成功状态
```

## 数据流与状态管理

### 角色变更数据流

```
用户点击下拉菜单 → 选择新角色
→ 发送 PUT /api/teams/{id}/members/{userId}/role
→ 后端验证权限 → 更新 team_members.role
→ 返回成功 → 前端更新成员列表（即时反馈）
```

### 移除成员数据流

```
用户点击"移除"按钮 → 弹出确认对话框 → 确认
→ 发送 DELETE /api/teams/{id}/members/{userId}
→ 后端验证权限 → 删除 team_members 记录
→ 返回成功 → 前端从列表移除该成员（即时更新）
```

### 删除团队数据流

```
用户点击"删除团队"小文字按钮 → 弹窗
→ 输入"我确认删除" → 点击确认按钮
→ 发送 DELETE /api/teams/{id}
→ 后端验证 Owner 权限 → CASCADE 删除
→ 返回成功 → 前端跳转到 /team 团队列表页
```

## 实现优先级

**阶段一优先级更高（核心功能）：**
- 用户加入团队是前提，没有成员就没有团队管理
- 阶段一完成后可以立即发布，用户开始使用核心功能
- 降低风险，分阶段实现便于发现问题及时调整

**阶段二依赖阶段一：**
- 阶段一完成后开发团队管理功能
- 所有 API 和 UI 在阶段二完成后统一测试
